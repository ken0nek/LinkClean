## Dependency notes

- The plans are **independent**; none blocks another. 003 (History before→after) is
  **built and staged as 1.2.2** (`07d8027`, unpushed) — the immediate ship is a version
  bump + submit, not new code.
- **004 (Safari extension v1) — Phase 0 gate PASSED** (2026-06-27, device-confirmed). Ken's
  Xcode-GUI handoff (target + App Group + `LinkCleanKit`) is done; the echo handler is now a
  real `URLCleaner` clean over `browser.runtime.sendNativeMessage`, proven on a physical
  device (sim Safari-extension behavior is unreliable). One finding: `nativeMessaging` is
  required in `permissions` (Apple's native-messaging doc) — it is the talk-to-own-container
  channel, not host access, so the `activeTab`-only privacy line still holds. **Phase 1**
  (popup Copy/Share/Markdown, the `safariCleanSucceeded` analytics case + test, ja/de popup
  localization, an in-app "enable the extension" pointer) is the remaining build.
- **001 ships the in-app surface only.** The high-value **action-extension** surface
  is deliberately deferred behind a latency/process-lifetime spike (a human
  measurement task, not executor work) — see 001 §"Surfaces" and "Maintenance notes".

## Shared context the executor needs

- iOS 26+, Swift 6.2, SwiftUI + SwiftData. **Default actor isolation = MainActor**
  (`SWIFT_DEFAULT_ACTOR_ISOLATION`); opt out with `nonisolated`.
- Local Swift package `LinkCleanKit/` — 4 layered targets, dependency direction
  compiler-enforced: `LinkCleanCore` (pure domain, no UIKit/SwiftData/network) →
  `LinkCleanData` (persistence, MainActor default) → `LinkCleanAnalytics` /
  `LinkCleanExtensionUI` / `LinkCleanIntents`. The app target is `LinkClean`.
- **Two-speed tests** (exact commands in each plan): kit fast lane runs on macOS in
  ~1s; app + extension tests run on an iOS 26.5 simulator (the app-test runner is
  known-flaky — `build-for-testing` is the reliable compile gate).
- Match existing patterns (`CLAUDE.md`). Each plan names the exemplar file to copy.

## Findings considered and rejected

- **ai-C inside the action extension as v1** — rejected for v1. Apple's on-device
  model is *system-managed / out-of-process* (so the extension's ~120 MB memory
  limit is **not** the blocker — see 001 §"Is Foundation Models available in action
  extensions?"), but running inference inside the short-lived share extension risks
  the process-lifetime + first-token-latency budget. LinkClean already decided this:
  `docs/product/ai-features.md:44,96,181`. Needs a measured spike before adoption.
- **iCloud sync / E4 as a Pro *headline*** — the free competitor (Clean Links) ships
  both free (`docs/strategy/competitor-clean-links.md:112-114,133`), so they are weak
  paywalls. E4 (002) is planned as an **opt-in utility** (still Pro-to-enable per the
  roadmap), not a headline; sync is not planned here.
